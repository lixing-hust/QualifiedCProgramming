# CAV Scope

本文件先定义：这个工具链适合处理什么样的 C 函数；然后再定义 Contract / Verify 两阶段的职责边界。

## 这个工具能处理什么样的程序

当前 CAV 工作流最适合处理下面这类程序：

- 单个目标函数，语义明确，输入输出边界清楚
- 命令式 C，控制流以顺序、分支、`for` / `while` 循环为主
- 算法题常见数据形态：
  - 标量
  - 数组
  - 字符串
  - 指针
  - 结构体 / 链表 / 树（前提是仓库里已有对应谓词和例子）
- 规格可以写成：
  - 输入约束
  - 输出关系
  - 内存拥有关系
  - 必要的数学定义或题目专用关系

最适合的题目类型：

- 纯计算函数
- 数组扫描 / 前缀处理 / 双重循环
- 字符串扫描 / 构造
- 读输入数组、写输出数组
- 在调用者提供的缓冲区上做原地更新
- 需要少量题目专用数学定义的算法题

## 不优先支持的写法

下面这些写法会显著提高 contract 和 verify 难度，应尽量避免：

- I/O、测试驱动、命令行主程序、平台依赖逻辑
- 大量宏、复杂条件编译、函数指针回调
- 依赖外部全局状态或隐藏副作用
- 在目标函数内部频繁 `malloc/free`
- 返回字符串字面量、返回静态内部缓冲区这类“语义是只读静态存储”的接口
- 把资源创建、资源填充、资源释放混在同一个待验证函数里

这不代表工具绝对不能处理，而是：

- 默认不应该把这类写法作为 contract 的首选输入形态
- 如果可以改写，应先改写成验证友好的接口

## 算法题的接口优先级

对算法题，默认按下面优先级设计验证输入：

1. 直接把需要读写的内存作为输入参数传入
2. 让函数在调用者提供的缓冲区、数组、工作区、`dp` 数组上工作
3. 默认不在待验证函数里 `malloc`，所需内存都应在调用前准备好并作为参数传入

优先把内存作为输入参数传入，原因很直接：

- `Require` / `Ensure` 更短，更稳定
- ownership 更清楚，不需要额外证明分配行为
- 更容易使用现成谓词：
  - `IntArray::full`
  - `UIntArray::full`
  - `CharArray::full`
  - `*_shape`
  - `*_seg`
- Verify 阶段更容易补 invariant 和 witness

因此，算法题里应遵守这条硬规则：

- 默认不要在待验证函数里 `malloc/free`
- 默认假定所有需要读写的内存都由调用者提供
- 需要的数组、缓冲区、工作区、辅助空间，都应写成输入参数
- Contract 阶段看到内部申请内存的原始题解时，应优先把接口改写成“内存从输入参数传入”的形式

需要以下内存时，优先改成输入参数：

- 输出数组
- 临时缓冲区
- 工作区
- 辅助数组
- `dp` 数组
- 字符串输出缓冲区

例如，和其写：

- `char *hello(void)` 然后返回新字符串或字面量

更推荐写成：

- `void hello(char *out)`，由调用者提供长度足够的输出缓冲区

再例如，动态规划题默认更推荐：

- `int climb(int n, int *dp)`

而不是：

- 在函数内部自己 `malloc` 一个 `dp` 数组

## 什么时候才使用 malloc

`malloc` 不是算法题验证输入的默认路线。只有在下面情况才例外考虑使用已规格化的 `malloc_*` 包装函数：

- 题目接口不能改，语义上必须返回新分配数组 / 新字符串
- 构造新对象本身就是题目语义的一部分
- 调用者不提供输出缓冲区，且你明确要验证“分配后返回”的行为

如果不满足上面这些条件，就不要使用 `malloc`，直接把内存作为参数传入。

即使使用 `malloc`，也不应直接在目标函数里裸写系统 `malloc` 作为首选；更稳的方式是：

- 先声明一个带规格的 `malloc_*` 函数
- 再在业务函数中调用它

## 两段流程

本目录采用严格的两段流程：

- Contract: 生成验证输入
- Verify: 验证规格输入

## Contract

Contract 负责：

- 从原始题意、原始代码和自然语言描述出发
- 判断当前接口是否适合验证
- 必要时把原始代码改写成验证友好的接口形态
- 生成验证友好的 `input/<name>.c`
- 必要时生成 `input/<name>.v`
- 写完整前后条件

如果原始接口不适合验证，Contract 应优先做的改写包括：

- 去掉 I/O
- 去掉测试脚手架
- 去掉平台依赖逻辑
- 把需要的内存改成输入参数
- 把“返回静态字面量/内部缓冲区”改成调用者提供输出缓冲区或显式分配对象

## Verify

Verify 负责：

- 消费 `input/<name>.c` 和可选 `input/<name>.v`
- 补 `Assert` / `Inv`
- 运行 symbolic / qcp
- 完成 manual proof

## 核心边界

- 重新设计接口与规格属于 Contract
- 验证与证明属于 Verify
- 不要在 Verify 中回退成“边修规格边验证”的混合流程
