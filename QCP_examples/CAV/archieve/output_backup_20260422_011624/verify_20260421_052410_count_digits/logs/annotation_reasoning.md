## Initial loop invariant plan

Program point: the only loop is `while (n > 0)` in `annotated/verify_20260421_052410_count_digits.c`. The current annotated file is identical to `input/count_digits.c` and has no `Inv`, so `symexec` would not have a stable relation between the original input `n@pre`, the shrinking loop variable `n`, and the accumulator `cnt`.

The postcondition is `__return == count_digits_z(n@pre) && emp`. The zero input branch returns before the loop, so the loop is reached only when the original input is positive. During the loop, `cnt` is the number of completed divisions by 10 and `n` is the remaining quotient. Unlike `digit_sum_z`, the supplied `count_digits_z` has the special value `count_digits_z(0) = 1`, so the invariant cannot simply be `cnt + count_digits_z(n) == count_digits_z(n@pre)` at all loop heads: after the final body execution, `n` becomes 0 and `cnt` is already the complete answer. Therefore the semantic invariant should split the two states: when `n > 0`, `cnt + count_digits_z(n) == count_digits_z(n@pre)`; when `n == 0`, `cnt == count_digits_z(n@pre)`.

The arithmetic safety facts should keep `0 <= n`, `n <= INT_MAX`, `0 <= cnt`, and `cnt + n <= n@pre`. Initialization has `cnt == 0` and `n == n@pre`. Preservation follows because under `n > 0`, `cnt + 1 + n / 10 <= cnt + n <= n@pre`, so `cnt++` stays within the integer range and the divided `n` remains nonnegative and bounded. At loop exit, the invariant plus `!(n > 0)` gives `n == 0` and the exact return relation. I will place a small `Assert` immediately after the loop fixing `n == 0`, `cnt == count_digits_z(n@pre)`, and the retained range facts, so the return witness consumes the postcondition directly.
