# EXPERIENCE 0407

这份笔记从 `leetcode/examples/` 里的完整样例中抽象出一组和具体题目无关的证明经验。重点不是某一题怎么过，而是哪些表示方式、annotation 结构和 proof pattern 在多类题里反复出现。

## 1. 规格分两层写：高层语义给用户，低层表示给实现

很多结构化题都同时存在：

- 一个适合对外表达功能正确性的抽象规格
- 一个适合函数体逐步执行的具体规格

最典型的是 BST：

- 高层用 `Bst::store_map`
- 低层用 `store_tree`

然后用

```c
high_level_spec <= low_level_spec
```

把两层连起来。

这样做的好处是：

- 客户端只看抽象语义
- 函数体证明只处理当前真正需要展开的表示
- 高层规格不会被底层局部更新细节污染

代表样例：

- `/home/yangfp/QualifiedCProgramming/QCP_examples/leetcode/examples/bst_insert/annotated/bst_insert.c`
- `/home/yangfp/QualifiedCProgramming/QCP_examples/leetcode/examples/bst_insert/coq/generated/bst_insert_proof_manual.v`

经验规则：

- 只要一个数据结构同时有“抽象语义谓词”和“具体表示谓词”，就优先分高低两层写，不要一开始把两种职责混在一个规格里。

## 2. 抽象谓词只在必要时展开，而且只展开到“刚好够当前语句”

多个样例都体现出同一个原则：

- 默认维持抽象谓词
- 只有在要读字段、写字段、分支判别、调用子函数时，才用 `which implies` 展开
- 展开后只暴露当前语句需要的那一层，不要把整棵结构全部打开

例如：

- `store_queue(q, l)` 只在入队/出队前展开成两条 `sll`
- `store_expr(e, e0)` 先只展开到 `store_expr_aux(e, e->t, e0)`，再按 tag 细分
- `store_tree(p, tr)` 只展开到“根节点 + 左右子树”

代表样例：

- `/home/yangfp/QualifiedCProgramming/QCP_examples/leetcode/examples/functional_queue/annotated/functional_queue.c`
- `/home/yangfp/QualifiedCProgramming/QCP_examples/leetcode/examples/eval/annotated/eval.c`
- `/home/yangfp/QualifiedCProgramming/QCP_examples/leetcode/examples/bst_insert/annotated/bst_insert.c`

经验规则：

- 如果你发现一个 `which implies` 把过多内部结构一次性全部暴露出来，通常说明展开过度了，后续 proof 会更脆。

## 3. 对“沿结构向下走”的程序，不变量最好用“上下文 + 当前焦点”

链表遍历、树搜索、分段数组循环，本质上都是“已经处理的上下文 + 当前还没处理的焦点”。

这类程序的不变量最稳定的写法通常不是直接描述“整个结构现在长什么样”，而是描述：

- 上下文如何保存
- 当前焦点是什么
- 把上下文和焦点重新拼回去后，仍等于目标整体语义

典型模式：

- 数组：`seg(prefix) * seg(rest)`
- 单链表：`lseg(prefix) * listrep(rest)`
- BST：`store_ptb(context) * store_tree(focus)`

代表样例：

- `/home/yangfp/QualifiedCProgramming/QCP_examples/leetcode/examples/int_array_merge_rel/annotated/int_array_merge_rel.c`
- `/home/yangfp/QualifiedCProgramming/QCP_examples/leetcode/examples/sll_auto/annotated/sll_auto.c`
- `/home/yangfp/QualifiedCProgramming/QCP_examples/leetcode/examples/bst_insert/annotated/bst_insert.c`

经验规则：

- 只要程序在“移动指针”或“移动区间边界”，优先寻找“context + focus”风格的不变量，而不是试图每步都直接描述整个结构。

## 4. 一个好的表示会把全局操作变成局部操作

很多验证难度不是来自程序本身，而是来自表示选错。

函数式队列是很好的例子。它不用“一个逻辑表直接对应一个物理链表”，而是规范成：

```c
l == app(l1, rev(l2))
```

这样：

- 入队变成对 `l2` 的头插
- 出队时只有在 `l1 == nil` 才做一次归一化

也就是说，一个合适的抽象表示能把“全局语义正确”转成“局部操作正确 + 一个很少发生的归一化步骤正确”。

代表样例：

- `/home/yangfp/QualifiedCProgramming/QCP_examples/leetcode/examples/functional_queue/annotated/functional_queue.c`

经验规则：

- 如果某个操作在当前表示下总要全局重写，先怀疑表示是否需要归一化，而不是先怀疑证明技巧不够。

## 5. tag 分支题先把“外层 tag”和“内层数据”分两步打开

AST、variant、union 一类题里，最容易犯的错误是：

- 一上来就同时展开 tag、字段和值语义

更稳的方式是两步：

1. 先从总谓词展开到“tag-aware 辅助谓词”
2. 再在具体 `case` 里展开该分支特有的字段

`eval` 里就是：

- `store_expr(e, e0)`
- `which implies store_expr_aux(e, e->t, e0)`
- 然后在 `T_CONST`、`T_VAR`、`T_BINOP`、`T_UNOP` 里分别细化

代表样例：

- `/home/yangfp/QualifiedCProgramming/QCP_examples/leetcode/examples/eval/annotated/eval.c`

经验规则：

- 只要结构是“tag + payload”，优先设计一个辅助谓词把这两层拆开，否则 case proof 会反复重复同样的展开工作。

## 6. 关系式证明要把“heap 状态”和“语义进度”同时放进 invariant

`safeExec` / relational 样例说明：

- 只写 heap 分段不够
- 只写抽象关系也不够

循环不变量必须同时携带：

- 当前堆形状，例如 `IntArray::seg(...) * IntArray::seg(...)`
- 当前语义进度，例如 `safeExec(... merge_from_mid_rel(...), X)`

这样每轮更新时，heap 变化和抽象语义推进才会同步。

代表样例：

- `/home/yangfp/QualifiedCProgramming/QCP_examples/leetcode/examples/int_array_merge_rel/annotated/int_array_merge_rel.c`
- `/home/yangfp/QualifiedCProgramming/QCP_examples/leetcode/examples/sll_split_while/annotated/sll_split_while.c`

经验规则：

- 如果题目里有显式 relation predicate 或 monadic spec，不要把它只放在函数前后置条件里，循环 invariant 往往也要携带“执行到了哪一步”的关系式状态。

## 7. `Assert` 最适合做阶段切换，不适合补救坏 invariant

从多个样例看，`Assert` 真正高效的用法是：

- 在一个循环结束后固定退出条件
- 在递归调用前把资源整理成子调用规格
- 在不同阶段之间切换表示

例如 mergesort 里，`Assert` 用来把“当前数组已被切成左右两段”的状态显式写出来，然后喂给递归调用。

代表样例：

- `/home/yangfp/QualifiedCProgramming/QCP_examples/leetcode/examples/int_array_merge_rel/annotated/int_array_merge_rel.c`

经验规则：

- 如果你想靠很多 `Assert` 来“修补”一个本来就不稳定的 invariant，通常应该回退去改 invariant，而不是继续堆 `Assert`。

## 8. 证明脚本经常分成三步：提纯事实、改写表示、最后算术或分类讨论

从 `sum`、`bst_insert`、`eval`、`int_array_merge_rel` 的手写证明里，可以看出很稳定的三段式：

1. `pre_process` / `Intros` / `prop_apply`
   把长度、非空、范围等纯事实抽出来
2. `sep_apply`
   用展开/折叠引理改写 heap 形状
3. `lia`、`destruct op`、`easy`
   收尾纯算术或有限分类

代表样例：

- `/home/yangfp/QualifiedCProgramming/QCP_examples/leetcode/examples/sum/coq/generated/sum_proof_manual.v`
- `/home/yangfp/QualifiedCProgramming/QCP_examples/leetcode/examples/bst_insert/coq/generated/bst_insert_proof_manual.v`
- `/home/yangfp/QualifiedCProgramming/QCP_examples/leetcode/examples/eval/coq/generated/eval_proof_manual.v`
- `/home/yangfp/QualifiedCProgramming/QCP_examples/leetcode/examples/int_array_merge_rel/coq/generated/int_array_merge_rel_proof_manual.v`

经验规则：

- 如果一个 witness 既有 heap 形状又有纯条件，先把这两部分分开处理；不要一开始就把所有目标都交给一个重 tactic。

## 9. 证明里反复需要长度事实，通常说明 annotation 应显式保留分段对齐关系

数组段、链表段、树路径 proof 里最常见的卡点之一是：

- 需要知道某段非空
- 需要知道某段长度正好是 0 / 1

手写证明里这常表现为反复使用：

- `IntArray.seg_length`
- `IntArray.full_length`
- 空段、单元素段、合并段引理

这背后的经验是：

- annotation 里应尽量保留边界变量间的对齐关系
- 例如 `k == p + (i - p) + (j - (q + 1))`
- 而不是只写“k 是某个当前位置”

代表样例：

- `/home/yangfp/QualifiedCProgramming/QCP_examples/leetcode/examples/int_array_merge_rel/annotated/int_array_merge_rel.c`
- `/home/yangfp/QualifiedCProgramming/QCP_examples/leetcode/examples/int_array_merge_rel/coq/generated/int_array_merge_rel_proof_manual.v`

经验规则：

- 如果证明里总要手工补“这个长度其实等于那个边界差”，就应该把这种边界关系提升进 invariant。

## 10. 当值语义不重要时，应主动降级到 shape-only 证明

不少样例成功的关键不是“把所有内容都证明出来”，而是明确知道什么时候不需要值语义。

例如：

- 数组初始化 / 拷贝骨架可以只用 `*_shape`
- 双链表遍历、反转骨架可先用 `dlistrep_shape`
- 某些局部旋转可以先只保 `store_tree_shape`

代表样例：

- `/home/yangfp/QualifiedCProgramming/QCP_examples/leetcode/examples/array_auto/annotated/array_auto.c`
- `/home/yangfp/QualifiedCProgramming/QCP_examples/leetcode/examples/dll_auto/annotated/dll_auto.c`
- `/home/yangfp/QualifiedCProgramming/QCP_examples/leetcode/examples/avl_insert/annotated/avl_insert.c`

经验规则：

- 如果用户要的是内存安全、结构保持、局部变换正确，而不是精确内容等价，优先选 shape-only 谓词，会显著减少证明负担。

## 11. 何时该回退到 annotation 层，而不是继续硬写 Coq

从这些例子总结，一个 witness 很难写时，优先判断它属于哪一种坏味道：

1. 需要不断手工恢复结构分解
   说明 annotation 展开点不对
2. 需要不断证明长度/边界显然事实
   说明 invariant 没把边界关系写全
3. 需要在多个 case 里重复相同展开
   说明缺少辅助谓词或 tag-aware 中间层
4. 需要在实现 proof 里直接表达高层抽象语义
   说明高低层规格没分开

经验规则：

- 一个 proof 难写，不一定是 tactic 不够强；很多时候是 annotation 和表示选得不够好。

## 12. 一个实用的默认检查表

开始新题时，可以先问自己：

1. 这题有没有“高层语义 / 低层表示”两层规格需求？
2. 当前程序是在操作“整体”，还是在维护“context + focus”？
3. 这题是否应该先选 shape-only 谓词？
4. 当前抽象谓词是否只展开到了当前语句需要的深度？
5. invariant 里是否同时保存了 heap 状态和语义进度？
6. 若 proof 卡住，是 tactic 问题，还是 annotation 少了边界关系或辅助中间层？

如果这 6 个问题先回答清楚，后续 annotation 和 Coq 证明通常会稳定很多。
